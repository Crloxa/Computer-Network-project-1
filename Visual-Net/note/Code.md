第六版编码测试（按手绘稿重制排布并扩展右下数据区）

输出分辨率：1330*1330  
实际分辨率：133*133  
最外圈：安全区 * 2（二维码外侧安全区）  
识别码：左上、右上、左下各一个 21*21 主定位块  
额外识别码：右下角预留一个 21*21 区域，保留 7*7 辅助定位块。该辅助定位块相对右下角向左、向上各偏移 1 个像素；辅助定位块四周保留 2 行 / 2 列安全区，最外侧 `row/col 130..132` 保持留白，其余单元并入数据区  

Header 区域：位于左上定位块右侧，范围为 `row 3..5, col 21..36`，总大小 3*16。  
Header 内部仍只承载 3 组语义不变的字段：  

1. 帧标志 + 尾长  
2. 校验码  
3. 帧号  

Header 回到最初的 `1bit = 1 个逻辑像素` 写法：3 个字段分别各占 1 行，每行 16 列，字段语义和位序保持不变，不新增协议字段。  

数据区共 4 块主区域，并在右下角额外追加一个 L 形扩展区域；编码时按从上到下、从左到右的区域顺序写入：  

1. 顶部数据区1-上半：`row 3..5, col 37..111`  
2. 顶部数据区1-下半：`row 6..20, col 21..111`  
3. 数据区2：`row 21..108, col 3..129`  
4. 数据区4：`row 109..111, col 3..129`  
5. 数据区3：`row 112..129, col 21..111`  
5. 右下扩展数据区：位于右下 `21*21` 预留区内，实际参与数据写入的单元构成一个 L 形区域：
   - 竖条：`row 112..129, col 112..120`
   - 横条：`row 112..120, col 121..129`

Header 缩回 `3*16` 后，顶部原本留给大块 Header 的空间也一并回填到数据区1。  
右下 21*21 保留区内，辅助定位块本体、其外侧 2 行 / 2 列安全区，以及最外缘 `row/col 130..132` 全部保持纯白。  

本阶段编码仍然使用黑白二值，不使用彩色三通道写入。  
所有数据区逻辑上合并为一个连续数据流，按 `顶部1-上半 -> 顶部1-下半 -> 2 -> 4 -> 3 -> 右下扩展区` 的区域顺序构造统一 cell 列表，再按行优先 bit 流连续写入。  
合并后总单元数为 `15028`，其中最后 `4` 个单元固定作为 padding 保持空白，因此有效数据单元数为 `15024`，可被 8 整除，对应逻辑容量为 `1878` 字节。  

当前阶段协议有效载荷仍保持 1242 字节，不扩容。  
如果数据区总容量超过 1242 字节，则剩余单元保持随机黑白乱码。  

调试预览：  

1. 默认正式输出仍然是黑白编码帧。  
2. 打开 `Layout_DEBUG` 后，会额外生成一张带彩色分区边界的预览图。  
3. 预览图文件名在正式帧文件名后追加 `_layout` 后缀，例如 `00000_layout.png`。  
4. 预览图只用于人工核对分区边界，不参与协议编码与解码。  

注意：

1. 这次改动只调整编码端排布。  
2. 解码端仍未适配新版 133*133 排布。  
3. 当前数据区2与数据区4使用 `col 3..129`，顶部和两侧收边后的空白保持为非数据区。  
4. 右下扩展数据区只在编码端生效，解码端后续需要按同样的 L 形顺序补齐读取逻辑。
