# 协议优化过程记录（V1 -> V1.6-108-4F）

## 文档目的
- 记录协议几何与解码鲁棒性优化过程。
- 说明每项优化改了什么，以及优化后有什么实际作用。
- 作为后续代码实现和联调验收的对照依据。

## 阶段 0：V1 基线（108x108, 3 Finder, 64-bit Header）
### 主要配置
- 逻辑网格：`108 x 108`
- 模块像素：`10`
- Finder：三角布局（左上、右上、左下）
- Header：`8 x 8 = 64 bits`

### 暴露问题
- 模块较大时，静区和几何余量紧张，边缘识别鲁棒性一般。
- 只有 3 个 finder，在强透视或局部遮挡时，几何解算稳定性受限。
- 64-bit header 冗余不足，头部轻微误码时恢复能力弱。

### 作用总结
- 优点是实现简单、兼容成本低。
- 适合通道较干净的场景，不适合复杂采集条件。

## 阶段 1：100x100 鲁棒尝试（module_px=9）
### 主要优化
- 逻辑网格改为 `100 x 100`。
- `module_px = 9`，有效区 `900 px`，边距 `90 px`（静区 `8.33%`）。
- Header 提升到 `160 bits`，引入镜像副本和 header CRC。

### 优化后有什么用
- 更宽静区显著提升边缘抗干扰能力。
- `9 px` 模块对采样更稳，finder 像素尺寸接近经验值。
- header 冗余后，头部误码不再直接导致整帧不可用。

### 代价
- 网格从 108 降到 100，和既有 `108x108` 生态存在配置分叉。

## 阶段 2：回归 108x108 并增强内部鲁棒设计
### 主要优化
- 保持 `108 x 108`，`module_px = 9`。
- 引入 timing 线与 alignment 块。
- Header 统一升级为 `160 bits` 冗余结构。

### 优化后有什么用
- 保留 108 网格兼容性，迁移成本低。
- timing 线可用于采样相位校正，降低位判决抖动。
- alignment 提升非线性畸变下的局部对齐能力。

## 阶段 3：V1.6-108-4F（当前推荐）
### 主要优化
- Finder 从 3 个升级为 4 个（四角布局）。
- 新增方向锚点：右下 finder 采用异形中心（`3 x 3` 反相）。
- 继续保留 `160-bit` 冗余 header、timing、alignment。

### 优化后有什么用
- 四角约束让单应矩阵估计更稳定，透视矫正误差更小。
- 异形角可在读 header 前确定方向，避免 90/180 度歧义。
- 在边缘裁切、旋转、轻微模糊等场景下，首帧锁定和连续解码稳定性更高。

### 容量影响
- 4 Finder 比 3 Finder 仅多占 `64 bits`（`8 bytes`）。
- 以当前口径，理论 payload 从 `1388 bytes` 降至 `1380 bytes`。
- 容量损失很小，但定位和方向判定鲁棒性提升明显。

## 阶段 4：规范补强与样例可视化（已落地）
### 主要优化
- 在 `V1.6-108-4F` 文档中补充“安全边框与容差”硬约束：外静区 `54 px = 6 modules`，采样抖动容差 `+-0.20 module`。
- 明确 timing 线比特序列与相位：`1010...`，并固定交点 `(24,24)` 为黑。
- 明确 payload 扫描和保留集合优先级，避免编解码器实现分歧。
- 明确 alignment 第二块的触发阈值：重投影误差 `> 0.35 module`，并同步容量重算。
- 生成并替换了结构化示意图：`bin/samples/sample_v1_6_108_4f_layout.png`（含区域标注与规格说明）。

### 优化后有什么用
- 安全边框和容差参数可直接用于摄像头端调参与验收，不再靠经验值。
- timing 相位统一后，跨实现（不同语言/团队）时采样结果一致性明显更高。
- payload 映射规则固定后，可避免“同一帧不同解码结果”的隐性兼容问题。
- 可视化样例可直接用于评审、联调和新成员理解协议结构。

## 阶段 5：仓库落地与提交记录（本次修改）
### 本次落地内容
- 按 `V1.6-108-4F` 思路改写了编码端常量与渲染逻辑，并已提交到当前分支。
- 目标改动包括：
  - 将 `module_px` 从 `10` 调整为 `9`
  - 将 3 finder 升级为 4 finder，并为右下角引入异形中心
  - 将 header 从 `64 bits` 扩展为 `160 bits`
  - 将 payload 扫描从“内区扫描”改为“全网格扫描 + 跳过保留区掩码”
  - 将 manifest 扩展为声明 `protocol_id` 与 `crc32_header`

### 本次修改过程
1. 先更新协议正文与参数文档，冻结 `V1.6-108-4F` 版式。
2. 再改编码端的常量、finder 绘制、header 打包和 payload 映射逻辑。
3. 同步调整文档目录结构、样例说明和 changelog，避免协议资料分散。
4. 使用 `clang++ -std=c++17 -Iinclude -Isrc -fsyntax-only ...` 做语法级检查。
5. 将实现结果提交到当前分支，形成 `优化补充` 与 `v1.6实现` 两次提交。

  - 要么协议固定只保留 1 个 alignment
  - 要么显式声明第二个 alignment 是否启用，不能由解码器临时猜测
- 当前主协议文档与过程草案需要持续同步，避免“设计草案”和“现行实现”漂移。

## 当前版本的综合价值
- 对现有 `108x108` 体系改动可控，兼容路径清晰。
- 误码、透视和旋转条件下的解码成功率预期更稳定。
- 便于后续叠加 FEC、交织、多帧投票等更高层容错机制。

## 建议的实现顺序
1. 先在编码端落地 4 Finder + 异形角绘制与保留区约束。
2. 在解码端增加异形角识别，完成方向归一化。
3. 接入 timing 相位校正与 alignment 局部修正。
4. 最后再调 `payload_max`、重复帧策略和 FEC 参数。
