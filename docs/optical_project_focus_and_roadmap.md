# 光学传输课程项目重点与下一步方向

> 参考页面：<https://whuangxmu.github.io/teaching/cni/project_view.html?id=optical>
> 更新时间：2026-03-06

## 1. 我认为的核心重点（从任务要求反推）

1. 这是一个“物理层可见光传输系统”而不是单纯图像处理作业。
2. 最终评价核心是**有效传输量**与**有效传输率**，不是只看是否能解码。
3. 必须遵守“时长约束”，不能通过拉长视频时长换吞吐。
4. 系统必须完整闭环：`二进制文件 -> 编码视频 -> 手机拍摄 -> 解码文件`。
5. 误码处理要分层：
   - 能检测错误（校验/CRC/FEC）。
   - 能标记错误（输出 `vout.bin`）。
   - 能统计错误（BER、丢失率）。
6. 工程上要可复现实验：固定输入、固定拍摄流程、固定统计脚本。

## 2. 关键信号与工程见解

1. 主要难点不在“编码成图片”，而在“抗拍摄扰动”：曝光、白平衡、滚动快门、透视畸变、帧丢失。
2. 编码方案应优先保障同步与可恢复性：
   - 起止帧标记
   - 帧序号
   - 每帧校验
   - 必要冗余（重发/FEC）
3. 颜色通道方案需考虑压缩损失（H.264 + yuv420p 会破坏细粒度色彩）。
4. 速率优化路径通常是：
   - 稳定识别率优先（降低误码）
   - 再逐步提高每帧承载位数和 fps
5. 评分第一指标是“第一段连续正确接收位”，意味着前段稳定性比尾部峰值吞吐更重要。

## 3. 与当前仓库现状的差距（可直接改进）

基于当前代码（`src/main.cpp`、`note/TODO.md`、`readme.md`）可见：

1. CLI 与题面接口不完全一致。
   - 题面建议：`encode in.bin out.mp4 max_ms`
   - 题面建议：`decode recorded.mp4 out.bin vout.bin`
   - 现状：主程序通过编译期常量切换 encode/decode，且 decode 没有 `vout.bin` 输出。
2. 平台耦合偏强。
   - 使用 `md / rd / del` 和 `.exe` 依赖，当前更偏 Windows 环境。
3. 错误标记输出能力不足。
   - 题目明确要求输出每位有效性标记，当前实现以恢复文件为主。
4. 自动化评测指标未标准化。
   - 缺少统一脚本来计算有效传输量、BER、丢失率、bps。
5. 稳定性风险点。
   - 视频解码阶段存在轮询等待与中间文件流程，性能与可维护性仍有优化空间。

## 4. 下一步方向（按优先级）

### P0：先满足题目硬性接口与可评测性

1. 拆分可执行入口：
   - `encoder`：`encode in.bin out.mp4 max_ms [fps]`
   - `decoder`：`decode recorded.mp4 out.bin vout.bin`
2. 明确 `vout.bin` 格式（建议）：
   - 每个输出 bit 对应 1 bit 标记（1=可信，0=错误/丢失）
   - 文件头写入版本号与长度，避免解释歧义
3. 新增评测脚本（Python/C++ 均可）：
   - 输入：`in.bin`, `out.bin`, `vout.bin`, `duration_ms`
   - 输出：有效传输量、总传输量、BER、丢失率、有效传输率

### P1：提升抗干扰能力（直接影响排名）

1. 增强帧同步：前导同步码 + 帧号 + 起止标识。
2. 纠错策略升级：
   - 每帧 CRC16/CRC32
   - 可选 RS/Hamming（按复杂度选）
3. 图像预处理增强：
   - 自适应阈值
   - 透视校正与亚像素定位
   - 曝光/白平衡归一化
4. 参数搜索：
   - 网格扫描 `fps × payload × block size × 校验强度`
   - 以“有效传输量最大”作为主目标

### P2：工程化与复现实验

1. 固化实验流程文档：拍摄距离、角度、光照、设备型号、分辨率、帧率。
2. 建立最小回归集：
   - 小文件（10KB）
   - 中文件（500KB）
   - 接近上限（10MB）
3. 输出报告模板：
   - 参数表
   - 指标表
   - 失败案例与原因归类

## 5. 建议的两周执行节奏（可直接照做）

1. 第 1-2 天：统一 CLI、定义 `vout.bin`、补评测脚本。
2. 第 3-5 天：完成同步码 + 帧号 + CRC 的稳定版本。
3. 第 6-8 天：加入透视校正和阈值自适应，跑第一轮参数扫描。
4. 第 9-11 天：加入轻量纠错（或重发策略），优化吞吐与误码平衡。
5. 第 12-14 天：固定最佳参数，整理复现实验与最终报告材料。

## 6. 交付检查清单

1. `encode/decode` 命令可直接运行并符合参数规范。
2. 任意测试输入都可产出 `out.bin` + `vout.bin`。
3. 评测脚本可自动输出 5 项关键指标。
4. 演示视频时长不超过用户设定时长。
5. 报告能解释“为什么这个参数组合最优”。
