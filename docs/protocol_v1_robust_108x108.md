# 协议 V1.6（鲁棒优化版）：108x108 四 Finder 网格

## 设计目标
- 保持 `108 x 108` 逻辑网格不变，降低对现有实现的改动。
- 通过 `module_px = 9` 与四角 finder 提升采样鲁棒性与定位稳定性。
- 在引入冗余 header、timing、alignment 后仍保留较高 payload 容量。

## 固定参数
- 物理输出尺寸：`1080 x 1080`
- `module_px = 9`
- 逻辑网格：`108 x 108` modules
- 数据区像素占用：`108 * 9 = 972 px`
- 四边边距：`(1080 - 972) / 2 = 54 px`
- 静区占比：`54 / 1080 = 5.00%`

## 安全边框与容差
- 外部安全边框（quiet zone）为 `54 px = 6 modules`（四边等宽）。
- 该值高于常用最小建议 `4 modules`，可容忍轻微裁切和边缘模糊。
- Finder 内侧额外保留 `1 module` 分隔带，避免定位块与数据区直接粘连。
- 解码采样点建议落在模块中心 `0.5 module` 处，允许抖动容差 `+-0.20 module`。

## Finder 与保留区
- Finder 尺寸：`7 x 7 modules`（像素为 `63 x 63`）
- 每个 finder 外围保留 `1` 模块分隔带
- Finder 保留块按 `8 x 8` 计算，四角布局：
  - 左上：`x=0..7, y=0..7`
  - 右上：`x=100..107, y=0..7`
  - 左下：`x=0..7, y=100..107`
  - 右下：`x=100..107, y=100..107`

### 方向锚点规则（避免旋转歧义）
- 左上、右上、左下使用标准 finder 图案。
- 右下使用异形 finder：外环与标准一致，中心 `3 x 3` 反相（黑白互换）。
- 解码时先识别异形 finder 所在角，再确定帧方向与坐标系。

## Header（160 bits）
- 建议尺寸：`16 x 10 = 160 bits`
- 建议位置：`x=8..23, y=8..17`
- 遍历顺序：row-major（先 y 后 x 的行扫描）

### 建议字段布局（鲁棒）
- bits `[0..63]`：主 header（V1 兼容核心字段）
  - `frame_seq`：16 bits
  - `payload_len_bytes`：16 bits
  - `crc32_payload`：32 bits
- bits `[64..127]`：主 header 镜像副本
- bits `[128..159]`：`crc32_header_128`（即 `CRC32(header_bits[0..127])`）

> 位序建议维持与 V1 一致：整数字段按大端位序写入，payload 内部按 `MSB-first`。

### 模式声明（强约束）
- `frame_manifest.tsv` 必须声明 `protocol_id = V1.6-108-4F`。
- 解码器在帧级识别失败时，禁止回退到 `100x100` 配置做盲解，避免误判。

## Timing 与 Alignment（新增）
- 水平 timing 线：`y=24, x=8..99`（长度 92）
- 垂直 timing 线：`x=24, y=8..99`（长度 92）
- 两条 timing 线交点：`(24,24)`，按 1 个模块计数
- Alignment 建议保留 1 个 `5 x 5` 块：`x=88..92, y=88..92`

### Timing 比特序列（强约束）
- 水平和垂直 timing 均使用交替序列 `1010...`。
- 起始相位定义：`(x=8, y=24)` 与 `(x=24, y=8)` 均为 `1`（黑）。
- 交点 `(24,24)` 固定为 `1`（黑），避免双线写入冲突。
- timing 线不参与 payload 扫描，也不参与 header 计算。

> 若后续验证表明透视畸变较大，可在 `x=20..24, y=88..92` 增加第二个 `5 x 5` alignment 块。

### Alignment 启用条件
- 默认启用 1 个 alignment 块（`x=88..92, y=88..92`）。
- 若四角单应矫正后的重投影误差 `> 0.35 module`，启用第二个 alignment 块（`x=20..24, y=88..92`）。
- 启用第二个 alignment 后，payload 理论容量减少 `25 bits`（约 `3 bytes`）。

## Payload 映射规则（强约束）
- 扫描顺序：在 `108 x 108` 全网格内按 row-major（`y` 递增，`x` 递增）扫描。
- 仅当模块不属于保留集合时，才写入或读取 payload bit。
- 保留集合优先级（从高到低）：
  1. 四个 finder 保留块（含分隔带）
  2. header 区
  3. 两条 timing 线
  4. alignment 区
- Payload 字节内位序：`MSB-first`。

## 容量与上限
- 总容量：`108 * 108 = 11664 bits`
- Finder 保留：`4 * 8 * 8 = 256 bits`
- Header：`160 bits`
- Timing 保留：`92 + 92 - 1 = 183 bits`
- Alignment 保留：`5 * 5 = 25 bits`
- 理论 payload：`11664 - 256 - 160 - 183 - 25 = 11040 bits = 1380 bytes`

### 第二 Alignment 启用时
- Alignment 保留变为：`2 * 25 = 50 bits`
- 理论 payload 变为：`11664 - 256 - 160 - 183 - 50 = 11015 bits = 1376 bytes`

### 实务建议
- 为提高解码稳定性，不建议打满理论容量。
- 推荐单帧 payload 上限：`960~1152 bytes`。
- 若通道噪声较高，可优先降至 `768~960 bytes` 并引入 FEC。

## 编码与解码建议
- 编码端：保持 `repeat = 3`（同帧重复发送）用于抗瞬时抖动。
- 解码端流程：
  - 四 finder 定位并识别异形角
  - 透视矫正到 `972 x 972`
  - 按 `9 x 9` 模块中心采样
  - 使用 timing 线做采样相位校正
  - 多帧 bit 多数投票（可选）
  - 校验 `crc32_header` 与 `crc32_payload`

## 与 V1 的关系
- 本方案可作为 V1 的鲁棒扩展（建议标记为 `V1.6-108-4F` 配置）。
- 若要兼容旧实现，需在帧头或清单中声明当前网格配置（`108` 或 `100`）。
