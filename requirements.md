### 为什么必须把 `01` 改成 `0xFF`？

我们来看评测机 `compare_files` 函数里的这段核心代码：
```c
fread(&rv, 1, 1, fv); // 从 vout.bin (fv) 里读取 1 个字节到 rv
for (size_t i = 0; i < 8; i++) {
    bv = rv & 1;      // 取 rv 的最低位（1 个比特）
    // ... 判断丢包率 ...
    rv >>= 1;         // rv 右移 1 位，检查下一个比特
}
```
**真相大白**：评测机是**按位（Bit）**来校验你的有效性的！
如果你输出 `0x01`（二进制是 `0000 0001`），评测机在这个循环里，只会认为这 1 个字节的**第 1 个比特是有效的**，而剩下的 7 个比特全被当作**丢失 (Lost)** 处理！这会让你的丢包率���升。

既然我们是“按字节”整体判断当前帧有没有丢，那么如果这一帧收到了，它里面的每一个字节的 8 个比特肯定都是有效的。所以，我们要填满二进制的 `1111 1111`，也就是十六进制的 **`0xFF`**。

### 第二步：完整的自测流程 (如何使用评测机)

为了跑通整个测试，你可以把老师提供的这套代码保存为一个单独的源文件，例如叫 `checker.c`，然后自己编译它并搭配你的 `encoder` 和 `decoder` 做端到端的自动化测试：

1. **编译评测机**：
   你可以直接用 gcc / g++ 或把这段代码放到一个新的工程里编译出一个 `checker.exe`（Linux/Mac下是 `checker`）。
   ```bash
   gcc checker.c -o checker
   ```

2. **生成原始测试文件**：
   利用评测机生成待编码的二进制文件。假设我们只测 1 组，文件大小为 102400 字节（100KB）：
   ```bash
   ./checker msg 1 102400
   ```
   > 运行后，当前目录会生成一个 `e1.bin` (这是我们要输入给 encoder 的原始文件)。

3. **运行你的编码器**：
   使用你刚刚编译出来的 `encoder` 生成视频，允许最长 10000 毫秒（10秒）：
   ```bash
   ./bin/encoder e1.bin e1.mp4 10000
   ```
   > 运行后，生成 `e1.mp4` 视频文件。如果是在真实���机测试环境下，这步就是把 `e1.mp4` 拿到手机上放，然后用另一个手机录下来存为 `recorded.mp4`。但为了自测解码逻辑，我们可以直接拿 `e1.mp4` 来解。

4. **运行你的解码器**：
   用你的 `decoder` 读取视频，并输出解码文件 `1.bin`（注意命名，评测机认 `%d.bin` 格式），以及有效性文件 `v1.bin`：
   ```bash
   ./bin/decoder e1.mp4 1.bin v1.bin
   ```
   > 运行后，会生成 `1.bin` 和 `v1.bin` 两个文件。

5. **运行评测机打分**：
   最后一步，让官方评测机对比 `e1.bin`（原文件）、`1.bin`（解码文件）、`v1.bin`（有效位文件），计算你的正确率：
   ```bash
   ./checker bench 1
   ```
   如果你的丢包率（Lost %）很低，且有效传输比特率（Val. b）很高，恭喜你，你的项目大功告成！完美符合验收标准！
